🏆 MASTER RULE: WHAT REVIEWERS ACTUALLY CHECK

Before sections, understand this:

A strong paper must prove:

✔ The problem is important
✔ There is a real research gap
✔ The contribution is clear and new
✔ The methodology is sound
✔ The evaluation is rigorous
✔ The claims are supported
✔ The limitations are acknowledged

If one of these is missing → rejection risk increases.

📌 SECTION-BY-SECTION RULE SET
1️⃣ TITLE — RULE SET
Must Follow:

Clearly indicate main contribution

Avoid vague words (Study, Analysis, Investigation)

Mention key technique if central (e.g., XAI, SHAP, Ensemble)

Avoid overclaiming (“Revolutionary”, “Perfect”, etc.)

Reviewer Questions:

Can I understand what this paper is about in 10 seconds?

Is it specific?

2️⃣ ABSTRACT — RULE SET (CRITICAL)
Must Include (In Order):

Problem context (1–2 sentences)

Research gap

Proposed method

Dataset description

Key numerical results

Main contribution

Must NOT:

Be vague

Skip results

Skip numbers

Be longer than allowed word count

Reviewer Thinking:

If abstract is weak → often reject without deep reading.

3️⃣ INTRODUCTION — RULE SET

This is the MOST important section.

Must Include:

Problem importance (economic, technical, social)

Current limitations

Clear research gap

Your proposed solution (brief overview)

Contributions (bullet points preferred)

Paper organization

Strict Rule:

You must explicitly state:

“The main contribution of this paper is…”

If contribution is unclear → rejection.

Avoid:

Long general AI history

Overcitation

No clear gap

4️⃣ RELATED WORK — RULE SET
Must:

Group papers by theme

Compare approaches

Mention limitations

Lead logically to your gap

Each reviewed paper should mention:

Method used

Dataset used

Results

Limitation

Final Paragraph Must:

Explicitly identify research gap.

Example:

“However, none of the above works integrate prediction, explanation, and actionable recommendations in a unified system.”

If no gap paragraph → weak paper.

5️⃣ METHODOLOGY — RULE SET

This section must be reproducible.

Must Include:

Dataset description (size, features, distribution)

Preprocessing steps

Model description

Mathematical formulation (if relevant)

Hyperparameters

Evaluation protocol (train/test split, validation)

Strict Rule:

Another researcher must be able to replicate your work.

If unclear → reviewer says:

“Methodology lacks sufficient detail.”

6️⃣ SYSTEM ARCHITECTURE (IF APPLICABLE)

If included:

Must:

Show architecture diagram

Describe modules clearly

Explain interaction between components

Avoid:

Turning paper into software documentation

Overemphasizing GUI

The research idea must dominate, not the interface.

7️⃣ RESULTS & DISCUSSION — RULE SET

This section determines acceptance.

Must Include:

Multiple evaluation metrics

Baseline comparison

Interpretation of results

Explanation of why results happen

Failure analysis

Strict Rules:

✔ Do not show only accuracy
✔ Do not show table without explanation
✔ Do not exaggerate improvements

Reviewer Asks:

Are improvements statistically meaningful?

Is comparison fair?

Are claims supported?

8️⃣ DISCUSSION — RULE SET

Often combined with Results.

Must discuss:

Trade-offs

Strengths

Weaknesses

Real-world impact

Computational complexity (if relevant)

Strong papers show balance.

Weak papers only praise themselves.

9️⃣ LIMITATIONS — RULE SET

Very important for maturity.

Must:

Admit weaknesses honestly

Mention dataset constraints

Mention generalization limits

Mention methodological limits

If you hide limitations → reviewer distrusts you.

🔟 CONCLUSION — RULE SET
Must:

Restate contribution

Restate key results

Emphasize impact

Mention future work

Must NOT:

Copy abstract

Introduce new ideas

1️⃣1️⃣ REFERENCES — RULE SET
Must:

Include recent papers (last 4–5 years)

Follow IEEE format strictly

Cite properly in text

Avoid blogs or non-academic sources

Minimum for strong paper: 15–25 references.

⚖️ OVERALL QUALITY RULES
Rule A: Novelty Clarity

If reviewer cannot answer:

“What is new here?”

Paper is weak.

Rule B: Evidence-Based Claims

Never say:

“This significantly improves…”

Without showing numbers.

Rule C: Balanced Tone

Avoid:

Emotional language

Marketing tone

Overconfidence

Academic tone only.

Rule D: Logical Flow

Each section must lead naturally to next.

Bad flow → reviewer fatigue.

Rule E: No Hidden Assumptions

Explain why:

You chose certain models

You chose certain metrics

You chose certain thresholds

🧠 REVIEWER ACCEPTANCE CHECKLIST

Before submission, answer YES to all:

 Is the research gap explicit?

 Is novelty clearly stated?

 Are results compared to baselines?

 Are claims supported with data?

 Are limitations acknowledged?

 Is methodology reproducible?

 Is IEEE format strictly followed?

If even one is weak → improve.

🏁 FINAL REVIEWER ADVICE

A paper gets accepted when it feels:

Clear

Necessary

Honest

Scientifically grounded

Logically structured

Not when it feels:

Impressive

Complicated

Long

Technically overloaded
